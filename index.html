Building OpenClaw: Automated Trading with OpenAlgo and a Custom API Layer

March 2025 8 min read Mohammed Fayaz

Algorithmic trading has always fascinated me — not just for the finance side, but for the engineering challenge. Getting low-latency systems to talk to broker APIs reliably, handling webhook signals without dropping orders, and building something that runs unattended while you sleep. That's the kind of problem that forces you to care about every layer of your stack.

OpenClaw is a project I built to scratch that itch. It sits between trading strategy signals and live broker execution, using OpenAlgo as the broker bridge and a custom REST API layer I built in Node.js to handle signal routing, validation, and order lifecycle management.

What is OpenAlgo?

OpenAlgo is an open-source algorithmic trading framework that acts as a unified broker bridge. It abstracts away broker-specific APIs (Zerodha, Fyers, Angel One, etc.) and exposes a clean, consistent REST interface you can call from any strategy — TradingView, Python, or your own backend.

Why not call the broker API directly? Broker APIs differ wildly — different auth flows, rate limits, error codes, and WebSocket formats. OpenAlgo normalizes all of that. It also handles session management and token refresh, which is a non-trivial problem in live trading.

The Architecture of OpenClaw

OpenClaw adds an intelligent routing layer on top of OpenAlgo. Here's the end-to-end flow:

architecture
  TradingView Alert  (webhook POST)
          ↓
  OpenClaw API  ←── validation, risk checks, signal enrichment
          ↓
  OpenAlgo Bridge  ←── broker abstraction, auth, normalization
          ↓
  Broker API  (Zerodha / Fyers / Angel One)
          ↓
  Exchange  (NSE / BSE)

Setting Up the OpenAlgo Connection

OpenAlgo runs locally or on a VPS and exposes endpoints like /api/v1/placeorder. Every request needs an API key generated from the OpenAlgo dashboard after linking your broker account.

javascript
// config/openalgo.js
const OPENALGO_BASE = 'http://127.0.0.1:5000/api/v1';
const API_KEY = process.env.OPENALGO_API_KEY;

async function placeOrder(payload) {
  const res = await fetch(`${OPENALGO_BASE}/placeorder`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${API_KEY}`
    },
    body: JSON.stringify(payload)
  });
  if (!res.ok) throw new Error(`OpenAlgo error: ${res.status}`);
  return res.json();
}

module.exports = { placeOrder };

Handling TradingView Webhooks

TradingView sends alerts as HTTP POST requests with a JSON body. You define the payload format in the alert message using TradingView's template syntax. A typical signal looks like:

json
{
  "symbol": "NIFTY25MARFUT",
  "action": "BUY",
  "quantity": "50",
  "order_type": "MARKET",
  "product": "MIS",
  "strategy": "supertrend_v2",
  "secret": "my-webhook-secret"
}

The OpenClaw webhook handler validates the secret, checks the symbol against an allowlist, enforces position sizing rules, then forwards the order:

javascript
// routes/webhook.js
router.post('/signal', async (req, res) => {
  const { symbol, action, quantity,
          order_type, product, strategy, secret } = req.body;

  // 1. Validate webhook secret
  if (secret !== process.env.WEBHOOK_SECRET)
    return res.status(401).json({ error: 'Unauthorized' });

  // 2. Symbol allowlist check
  if (!ALLOWED_SYMBOLS.includes(symbol))
    return res.status(400).json({ error: 'Symbol not permitted' });

  // 3. Risk check — max qty per signal
  const qty = parseInt(quantity);
  if (qty > MAX_QTY_LIMIT)
    return res.status(400).json({ error: 'Quantity exceeds limit' });

  // 4. Forward to OpenAlgo
  try {
    const result = await placeOrder({
      apikey: process.env.OPENALGO_API_KEY,
      strategy, symbol, action: action.toUpperCase(),
      exchange: 'NFO', pricetype: order_type, product, quantity: qty
    });
    res.json({ success: true, orderId: result.orderid });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

MinMax Signal Integration

One of the strategies wired into OpenClaw uses a custom MinMax indicator — it identifies local price minima and maxima over a rolling window and fires signals at potential reversal zones. A scheduled job fetches OHLC data, runs the MinMax calculation, and posts to the OpenClaw webhook if a threshold is crossed.

javascript
// strategies/minmax.js
function detectMinMax(closes, window = 5) {
  const last = closes.length - 1;
  const slice = closes.slice(last - window, last + 1);
  const mid = slice[Math.floor(window / 2)];

  const isMax = slice.every((v, i) =>
    i === Math.floor(window / 2) || mid >= v);
  const isMin = slice.every((v, i) =>
    i === Math.floor(window / 2) || mid <= v);

  if (isMax) return 'SELL';
  if (isMin) return 'BUY';
  return null;
}

What I Learned

  • Webhook idempotency matters — TradingView can fire duplicate alerts. Always deduplicate by signal ID or timestamp before placing an order.
  • Fail loudly and fast. A silent failure in a trading system is worse than a crash. Log everything, alert immediately.
  • Secrets management is non-negotiable. Broker API keys carry real financial risk — environment variables only, never hardcoded.
  • Rate-limit your own webhook endpoint to prevent runaway strategy bugs from placing hundreds of orders in seconds.

OpenClaw is a personal project — not financial advice, not a production trading product. But it's been one of the most satisfying systems I've built because every bug has real, immediate consequences. Nothing sharpens your error handling instincts like money on the line.

Docker to Kubernetes: A Backend Engineer's Practical Migration Guide

February 2025 10 min read Mohammed Fayaz

Most backend engineers reach Docker naturally — you containerize your app, write a docker-compose.yml, and life gets better. Reproducible environments, no more "works on my machine," easy local dev setup. Docker is genuinely great for what it does.

Then your app grows. You're running multiple services, need zero-downtime deployments, want automatic restarts when a container crashes, and traffic that scales horizontally on load spikes. That's when Docker alone starts to feel limiting — and Kubernetes starts making sense.

This is a practical walkthrough of containerizing a NestJS API with Docker, then migrating that setup to Kubernetes. I'll focus on concepts that matter day-to-day, not the full K8s spec.

Step 1: Containerize the NestJS App

Start with a clean, production-ready Dockerfile using a multi-stage build — the build stage compiles TypeScript, the runtime stage only ships what's needed to run. This keeps the final image lean.

dockerfile
# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Runtime
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./
ENV NODE_ENV=production
EXPOSE 3000
CMD ["node", "dist/main"]

Step 2: Docker Compose for Local Dev

A solid docker-compose.yml is your local development environment and a useful reference for everything your app needs to run:

yaml
version: '3.9'
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/appdb
      - JWT_SECRET=${JWT_SECRET}
    depends_on:
      - db

  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:

Step 3: Understanding What Kubernetes Actually Does

Kubernetes is a container orchestrator. It manages where containers run, restarts them when they crash, balances traffic across instances, and handles rolling deployments. The mental model shift from Docker:

  • Docker runs containers. K8s decides which node runs them, how many replicas exist, and what happens when one dies.
  • You describe desired state in YAML manifests. K8s continuously reconciles reality to match that desired state.
  • There are no manual docker run commands in production K8s. Everything is declared.

Core K8s primitives: Pod — smallest unit, wraps one or more containers. Deployment — manages a replica set, handles rollouts and rollbacks. Service — stable network endpoint that routes traffic to pods. Secret / ConfigMap — inject config and credentials without hardcoding them.

Step 4: The Deployment Manifest

A Deployment tells K8s to run 3 replicas of your container, use a specific image, and restart pods if they become unhealthy — confirmed via a readinessProbe.

yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nestjs-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nestjs-api
  template:
    metadata:
      labels:
        app: nestjs-api
    spec:
      containers:
        - name: api
          image: your-registry/nestjs-app:latest
          ports:
            - containerPort: 3000
          envFrom:
            - secretRef:
                name: app-secrets
          resources:
            requests:
              memory: "128Mi"
              cpu: "100m"
            limits:
              memory: "256Mi"
              cpu: "500m"
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 10
            periodSeconds: 5

Step 5: Exposing the App with a Service

Pods are ephemeral — they get new IP addresses every restart. A Service gives you a stable DNS name and IP that always routes to healthy pods. For external traffic, use a LoadBalancer type:

yaml
apiVersion: v1
kind: Service
metadata:
  name: nestjs-api-svc
spec:
  selector:
    app: nestjs-api
  ports:
    - protocol: TCP
      port: 80
      targetPort: 3000
  type: LoadBalancer

Step 6: Secrets and Zero-Downtime Deploys

Never put secrets in your image or manifests. Use K8s Secrets, injected as environment variables at runtime:

bash
kubectl create secret generic app-secrets \
  --from-literal=DATABASE_URL="postgres://..." \
  --from-literal=JWT_SECRET="your-secret"

A rolling deployment updates pods one at a time, only terminating old ones once new pods pass their readiness check. Update the image tag and apply:

bash
kubectl set image deployment/nestjs-api \
  api=your-registry/nestjs-app:v2

kubectl rollout status deployment/nestjs-api

# If something goes wrong:
kubectl rollout undo deployment/nestjs-api

When to Use Kubernetes (and When Not To)

Kubernetes adds real operational overhead. It's worth it when you have multiple services, need horizontal autoscaling, or require zero-downtime deploys with rollback capability. For a single-service app on one server, Docker + NGINX + PM2 is often the right call — I use this on AWS EC2 for smaller deployments and it works well.

The mental model I use: if you need to answer "how do I restart this automatically when it crashes and route traffic away from it while it's down" — that's a K8s problem. Everything else is probably fine with Docker Compose.

The jump from Docker to Kubernetes feels steep at first because of the YAML volume and new abstractions. But the core loop is simple: write a Deployment, wire up a Service, manage config with Secrets. Everything else is optimization on top of that foundation.
